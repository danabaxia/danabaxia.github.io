AI-Assisted Manufacturing Test Validation

Summary

How we built a three-mode validation system that lets an AI agent and a developer validate manufacturing test changes on a laptop — no hardware, no lab time, no queue.

The Problem

Manufacturing test stations for battery products (Enphase IQ Battery R4) run a complex pipeline: TRD Excel documents define test flows, a Python framework parses them into JSON artifacts, test steps communicate with hardware over CAN bus, and results determine pass/fail at the production line.

Every change to this pipeline — a new firmware update step, a modified spec limit, a refactored driver — carries risk. The traditional validation path requires a physical test station with real hardware: PCAN USB adapters, battery management units, power conversion units, DC supplies, relays. Lab time is scarce. Engineers queue for bench access. A single broken import path discovered at the station wastes hours.

The Core Question

How much of this validation can an AI agent and a developer do together on a laptop, before anything touches a production station?

The Approach: Three Validation Modes

Rather than building one monolithic validator, we designed a confidence ladder — three modes that validate at increasing levels of fidelity, each catching a different class of defect.

Mode 0 — TRD Parse + Logic Validation

The cheapest and fastest check. Mode 0 takes the TRD Excel file (the source of truth for every test station), runs it through the real framework parser, and validates the generated JSON artifacts for internal consistency.

It answers: Can the framework parse this TRD correctly, and are the generated data files logically sound?

What Mode 0 Catches
  • Missing or duplicate resource/variable/channel definitions
  • Forward references in dependency graphs (step 8 depends on step 12, which hasn't run yet)
  • Cycles in must-pass chains
  • Skipped steps with contradictory flags (retries on a step that never executes)
  • Import path portability issues (mtt_01000.RWK.* vs RWK.*)
  • Framework version mismatches

Mode 0 runs 30+ checks across two phases: cross-file consistency (C-00 through C-14) and flow logic (L-01 through L-19). It uses phase gating — if the TRD can't parse, downstream checks automatically SKIP rather than producing misleading results.

A typical run takes under 10 seconds and produces a JSON report with machine-readable check IDs, source references pointing to exact file paths and JSON keys, and a clear PASS/FAIL/WARN/SKIP status for each check.

Mode 1 — Virtual Driver + State Machine Validation

Mode 1 operates at the driver level. The PCAN CAN driver (pcan.py) supports two modes: physical (real hardware) and virtual (golden unit simulation). In virtual mode, the driver intercepts 20 hardware-bound primitive methods and returns profiled responses from a golden unit behavior file — a static JSON document that defines what a known-good battery system looks like in idle, charge, and discharge states.

This is where the battery state machine lives. Rather than tracking charge state as a raw string (self._virtual_battery_state = "charge"), we formalized it into a proper state machine with:

  • 5 states: IDLE, CHARGING, DISCHARGING, SHUTDOWN, ERROR
  • Validated transitions: SHUTDOWN can only go to IDLE or ERROR, not directly to CHARGING
  • Per-device tracking: each serial number has its own state, history, and transition log
  • Strict/non-strict modes: strict raises on invalid transitions (for testing), non-strict warns but proceeds (for production compatibility)
  • Trigger logging with intent: set_crequest:+0.300 tells you a discharge command was issued at 30% power, not just "some function called"
Sync Contract

The state machine is the source of truth. A legacy string mirror (_virtual_battery_state) stays in sync through a single _sync_battery_state() helper — one mutation path, no direct writes to the string.

The validation suite for this layer runs 94 checks across 18 test groups:

Mode 2 — Framework GUI Validation (Production-like)

The full framework with real test sequencer, real step instantiation, real result collection — but running against virtual instruments instead of physical hardware. This is the closest to production without a station. It requires the most setup and runs the slowest, but catches integration issues that only surface when the full framework lifecycle executes.

How AI Fits In

The AI agent (Claude Code, running as a CLI tool) participates at every level:

Planning and Architecture

The three-mode design, the state machine transition map, the sync contract rules, the validation check inventory — all developed through conversation between engineer and AI. The engineer brings domain knowledge (what CAN bus edge cases matter, which framework calls require Qt), the AI brings structure (formal state machines, exhaustive check matrices, phase gating logic).

Implementation

The AI writes code directly in the codebase — editing pcan.py, creating state_machine.py, extending validate_system.py with new test groups. It reads existing code before modifying it, runs the validation suite after changes, and iterates on failures.

Validation Execution

The AI runs Mode 0 and Mode 1 validation, reads the full output, and reports results. When a check fails, it can trace the failure to a specific file and line. When a change is proposed, it can predict which checks will be affected.

Review and Documentation

After code changes, the AI updates the SOP, the project status document, and the CLAUDE.md architecture reference. Every validation run is logged to virtual_runner/logs/ with timestamps, durations, and full stdout/stderr capture — reviewable by both the next engineer and the next AI session.

What We Learned

Separate your validation layers

Key Insight

The single most valuable decision was not trying to validate everything at one level. Mode 0 catches 80% of TRD issues in 5 seconds. Mode 1 catches driver/state issues in 30 seconds. Mode 2 catches framework integration issues in 2 minutes. Each layer has a clear scope and a clear failure taxonomy.

Golden profiles are virtual oracles

The golden_unit_behavior.json file is not just test data — it's the definition of what "correct" looks like. It declares device topology, telemetry values for three battery states, cell-level measurements, firmware versions, and transport modes. Every virtual test runs against this profile. When you modify the profile, you're modifying the spec.

Building a BAD profile (golden_unit_behavior_BAD.json) with deliberately out-of-spec values enables negative testing: verify that the system correctly flags violations, not just that it accepts good data.

State machines prevent state bugs

The original battery state tracker was three lines of string assignment scattered across two functions. It worked — until you needed to answer "what transitions happened?" or "is this transition valid?" or "did the charge loop create 500 history entries from repeated calls?"

Formalizing it into a state machine with an enum, a transition map, and a history log made the system debuggable and testable. The same-state no-op guard (calling set_crequest(-0.5) ten times creates one transition, not ten) was a direct result of thinking about the charge loop's real behavior.

Sync contracts catch desynchronization

Anti-Pattern

When you have two representations of the same state, never write to both independently. Define a sync contract: one is source of truth, the other is a mirror, and there is exactly one function that updates both.

T18 tests this explicitly — five checks that verify the SM and the string agree after every operation.

Log everything for the next session

AI sessions are stateless. The next conversation starts fresh. If the validation output only existed in the terminal, it would be lost. Writing every run to virtual_runner/logs/ with a timestamp means the AI in the next session can read the last run's results, understand the current state, and pick up where we left off.

Let the AI run the tests

Fastest Feedback Loop

The fastest feedback loop is when the AI can run validation immediately after making a change. No context switching, no copy-pasting commands, no waiting for a human to press enter. The validation commands are documented in the SOP, the paths are absolute, and the output is structured. The AI runs 94 checks in 30 seconds and reports the result in the same conversation.

Portability

The validation system is portable — virtual_runner/ copies to any project in the mono-repo. Edit one config file (ticket number and TRD filename), run run_validation.bat mode0, and you have a full TRD validation for a new project in under a minute.

What's Next

  • Mode 2 (v2.0): Framework-lite headless runner that exercises the real test sequencer without the GUI. Architecture is designed, implementation is planned.
  • Physical hardware validation pass: Connect real R4 units, compare virtual vs physical telemetry output, and calibrate the golden profile
  • Per-device state tracking in the driver: The state machine supports it, the driver currently uses global state only. Wiring per-device tracking enables multi-unit simulation
  • CI integration: Run Mode 0 + Mode 1 on every commit. Block merges on FAIL
